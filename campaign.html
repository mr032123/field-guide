Backup demo — automated campaign pipeline

One agent runs audience research → brief → territories → compliance scoring → recommendation, for any persona. The live moment: switch the persona, run it again. · Main build guide

What this is and when to build it

This is the backup for the measurement demo — decide at your Day-4 gate. It shows an agent running a complete campaign development process for an invented ETF, grounded in your real templates and guidance documents, ending in a campaign book (Word), a deck (PowerPoint), and a hero visual (Copilot image). The demo's peak is the audience switch: same process, same controls, different persona — one changed word in the instruction.

What it proves The pipeline is the cost story (a campaign process that runs itself, with legal and identity guidance enforced by an agent, not a review queue). The persona switch is the opportunity story (personalization as a built-in capability). It reuses the same five personas as the measurement work — one audience science powering both measuring and making.

Feasibility test — four checks, ~2 hours, before committing

  1. Image generation (already confirmed working): in Copilot chat, ask for an image. Done — yours generated.
  2. Agent grounding: in Copilot Studio, create a throwaway agent, add 2–3 of your files as Knowledge (the product sheet and one persona card), publish, then ask it in the test pane: "What is the product in the product sheet, and what does it cost?" If it answers from the file — grounding works. If it answers generically or says it doesn't know, the files haven't indexed yet: wait 15 minutes and re-ask before concluding anything.
  3. Word export: the plan is to run the agent inside the Microsoft 365 Copilot app (the main Copilot chat, where your agents appear in the right-hand panel or under "Agents" once published). Run any prompt at your throwaway agent there, and look at the buttons under its response for an export or "Edit in Word / Edit in Pages" option. If it exists, the campaign book becomes a Word file in one click, no copying. If it doesn't exist in your tenant, use the fallback in step 2.1.
  4. Deck from document: open PowerPoint → open your deck template file → click Copilot in the ribbon → look for "Create a presentation from a file" → point it at any Word document you have. If it builds slides, the deck chain works.

All four pass → this backup is real and the 2-day estimate holds. Checks 3 or 4 fail → the demo still works, ending at the campaign book + image (skip the deck), which is still a complete story.

1 · Build the agent

1.1 · Assemble the knowledge folder
30 min · one folder, clear filenames
›

Create a folder in OneDrive or SharePoint named Campaign Studio files. Put copies of these in it (copies — leave your originals wherever they live), renamed so their purpose is obvious to the agent:

  • PRODUCT SHEET - [invented ETF name].docx — the fake product sheet
  • PERSONA - Amy.docx … PERSONA - Linda.docx — one file per persona card
  • AUDIENCE RESEARCH.docx — the corroborating research
  • CREATIVE BRIEF TEMPLATE.docx
  • LEGAL GUIDANCE.docx — the rules creative must follow
  • IDENTITY AND TONE GUIDELINES.docx — your voice/visual guidance
  • MEDIA CONTEXT.docx and MEASUREMENT CONTEXT.docx

Filenames matter: the agent's instructions refer to files by these names. If a document is a deck (.pptx), that's fine — agents can read those too — but Word/PDF index most reliably. Keep the folder under ~20 files; more dilutes retrieval.

1.2 · Create the agent
30 min · Copilot Studio, same as your other agents
›

copilotstudio.microsoft.com → Agents → New agent → skip to the configure form. Name: Campaign Studio. Model: the default is fine. Web search: OFF — different reason than the measurement build: this agent must build from YOUR files (product truth, your guidance), not from the internet's idea of marketing. In Knowledge, add every file from the folder (Add knowledge → browse to the folder → select all the files). Paste the instructions below, then Publish — and when publishing, enable it for Microsoft 365 Copilot and Teams (there's a channels/availability step in publishing; tick both), because the demo runs in the Copilot app where the Word-export button lives.

You are Campaign Studio, an automated campaign development pipeline.

When the user sends: "Run the pipeline for [product] for [persona]"
(persona is one of: Amy, Brad, Emily, Jenny, Linda), execute all
five stages below in one single response, in order, with a clear
heading per stage. Do not ask questions first. If the user adds
"concise mode", write stages 1-3 at half length.

STAGE 1 — AUDIENCE INSIGHT
From the PERSONA file for the named persona and the AUDIENCE
RESEARCH file: who this person is, what they want, what stops them,
where to reach them — and the single insight about them most
relevant to this product. 150 words maximum.

STAGE 2 — CREATIVE BRIEF
Follow the section structure of the CREATIVE BRIEF TEMPLATE file:
objective, audience, single-minded proposition, support, tone (from
the IDENTITY AND TONE GUIDELINES file), mandatories (from the LEGAL
GUIDANCE file), recommended channels (from the MEDIA CONTEXT file,
matched to this persona's media habits).

STAGE 3 — CAMPAIGN TERRITORIES
Three genuinely distinct creative territories. For each: a name,
the big idea in two sentences, one example headline, one example
social post line, and the two best channels for this audience.

STAGE 4 — COMPLIANCE AND CONSISTENCY SCORECARD
Score every territory against the LEGAL GUIDANCE and IDENTITY AND
TONE GUIDELINES files. Output a table: rows are the specific rules
or criteria drawn from those two files; columns are the three
territories; each cell is PASS, FLAG, or FAIL. Below the table,
one line of explanation for every FLAG or FAIL.

STAGE 5 — RECOMMENDATION AND IMAGE BRIEF
Name the winning territory and justify it in three sentences,
citing the scorecard. Then write "IMAGE BRIEF:" followed by one
paragraph an image tool could render as the hero visual: subject,
setting, mood, style, colors. No real people, no logos, no text
inside the image.

Rules: ground every stage in the knowledge files; where a file
contradicts your general knowledge, the file wins. Never invent
product facts that are not in the PRODUCT SHEET file. Keep the
full response under 1,200 words.
1.3 · Test and tune
1–2 hrs · run it, read hard, adjust
›

In the agent's test pane, type: Run the pipeline for [your invented ETF's name] for Emily. Read the output against four tests:

  • Grounding: does Stage 1 sound like YOUR Emily (details from her card), or a generic 38-year-old? Does the brief use your template's section names? Generic output = knowledge files not indexing or not named as the instructions expect — check filenames match.
  • Product truth: every product claim traceable to the fake product sheet. If it invents features, add the missing fact to the product sheet rather than fighting the agent.
  • Scorecard teeth: Stage 4's criteria should be recognizably from your actual guidance documents, and at least one territory should FLAG something — a scorecard where everything passes looks fake. If all-pass, your guidance files may be too vague for the agent to extract rules from; add a "key rules" summary section at the top of the legal guidance copy.
  • The switch: run Run the pipeline for [product] for Linda. Stages 1–3 should be visibly, meaningfully different — different insight, different tone, different channels, different territories. This difference IS the demo; if the two runs feel samey, sharpen the persona files' distinctive details.

Tune by editing the instructions or the files, re-publish, re-run. Two or three cycles is normal.

2 · Document, deck, and image

The agent's chat response becomes three artifacts, all inside Microsoft: a Word campaign book, a PowerPoint deck, and a rendered hero visual. This chain is what answers "can it make a deck" — the agent writes, the suite renders.

2.1 · Campaign book → Word
1 click if export exists · fallback included
›

Primary path: run the pipeline in the Microsoft 365 Copilot app (not the Studio test pane): open Copilot → find Campaign Studio in your agents list (it appears there because you published to Microsoft 365 Copilot in 1.2) → send the run command → when the response finishes, use the export / "Edit in Word" control under the response. Name the file Campaign book - [persona].docx, save it into a folder called Campaign outputs. One click, no copying.

Fallback if your tenant has no export button: select the full response in the chat → copy → open your brief template in Word → paste. It's one paste, inside your own suite, and takes fifteen seconds — acceptable for a backup demo, and invisible if you do it as a pre-run before the meeting. (For the live switch moment, you'll show the response on screen directly, so no export is needed live either way.)

2.2 · Campaign book → deck
1 click in PowerPoint · template first
›

Open PowerPoint → open your deck template file first (Copilot builds slides in whatever theme is open — opening the template first is how you keep your look) → click Copilot in the ribbon → choose Create a presentation from a file → select Campaign book - [persona].docx. Review the slides it builds; expect to delete a couple and tighten titles — five minutes of cleanup, not authorship. Save as Campaign deck - [persona].pptx in the same outputs folder.

2.3 · Image brief → hero visual
2 min · in Copilot chat, where your image test worked
›

From Stage 5 of the campaign book, copy the paragraph after "IMAGE BRIEF:" → open Copilot chat (the same place your test image generated) → paste it with this prefix: Generate an image: → generate. If the first render misses, add one steering line ("more minimalist," "warmer light") rather than rewriting the brief. Save the image into the outputs folder as Hero visual - [persona].png.

Labeling rule: anywhere this image appears (deck, demo), caption it "AI concept visual — not final creative." Financial-promotion review will ask; having the label already there is the right answer.

3 · Demo script — 6 to 8 minutes

3.1 · Pre-run everything for persona A
day before · nothing live depends on generation speed
›

Run the full chain for your first persona (pick the one whose output read best in testing): campaign book, deck, hero visual, all saved in the outputs folder. These are your show-and-tell artifacts. Also screen-record one full pipeline run — that recording is your backup if anything misbehaves live.

3.2 · The beats, in order
rehearse twice, timed
›
  1. The artifacts (2 min): open the persona-A deck and page through: brief → territories → scorecard → recommendation → hero visual. One sentence of setup: "One instruction produced this — research, brief, territories, a compliance scorecard against our actual legal and identity guidance, a recommendation, and the concept visual."
  2. Pause on the scorecard (30 sec): "The rules in these rows are our real guidance documents. The reviewer here is an agent — the controls are in the tool, not in a queue."
  3. The switch (3–4 min, live): open Copilot, the Campaign Studio agent, and type the run command with a different persona and concise mode: Run the pipeline for [product] for Jenny. Concise mode. Say nothing while it streams. As the stages appear, point at what changed: different insight, different tone, different territories, same guardrails. Land the line: "Same process, same controls, different human. The change was one word."
  4. Optional applause beat (1 min): copy the new run's IMAGE BRIEF paragraph into Copilot image generation and let persona B's visual render in the room. Only do this if it behaved in rehearsal — a good demo ends on a peak, not on a spinner.
3.3 · Questions to be ready for
the three that will come
›
  • "Is this creative any good?" — "It's a first-pass system: the territories are starting points a team refines. The point isn't that it replaces creatives; it's that research, briefing, compliance checking, and audience adaptation stopped being weeks of coordination."
  • "Would legal actually sign off on agent-scored compliance?" — "The agent applies the written rules and flags; humans still approve. It moves legal from reading everything to reading flags — that's the saving."
  • "How long did this take to build?" — answer honestly with your build days; the small number IS the point.

Troubleshooting

Output is generic — doesn't sound like your files. Knowledge hasn't indexed (wait 15 minutes after adding files, then re-test) or filenames don't match what the instructions reference (compare exactly). Re-publish after any change.
It invents product facts. The fact it needed isn't in the product sheet. Add the fact to the sheet rather than adding "don't invent" scolding to the instructions — agents follow available truth better than prohibitions.
The scorecard passes everything. Your guidance files are too narrative for rule extraction. Add a section at the top of the legal guidance copy titled "Key rules" with 6–10 one-line rules; the scorecard will use them as rows.
Two personas produce similar campaigns. The persona files lack distinctive, concrete detail where it matters (what they'd never respond to, words they'd use, channels). Sharpen the two cards you demo with; re-run.
Response cuts off mid-stage. Output length limits. The instructions cap at 1,200 words — if it still truncates, run with "concise mode", or split the run: "Run stages 1-3…" then "Run stages 4-5 for the same product and persona."
No export/Edit-in-Word button in Copilot. Use the fallback in 2.1 (one paste into your Word template). For the live switch this doesn't matter — the streaming response on screen is the demo.
"User does not have sufficient permissions in Microsoft Dataverse for flow…" when the agent calls a flow tool. The agent and flow are fine — this is the M365 Copilot app refusing to bind the connection between them. Work in order: (0) Publish the agent — if you added the flow as a tool and only tested, surfaces outside Studio are running the last published version, and a stale registration can produce misleading connection errors; republish, then retry Manage connections → Refresh → Connect. (1) Test the agent in Copilot Studio's test pane — same environment as the flow, friendliest permissions; if it works there, demo from Studio or Teams. (2) Confirm the agent and the flow live in the same environment (name shown top-right in Studio) — if not, rebuild the flow in the agent's environment. (3) On Manage connections: select the row → Refresh → Connect again (sometimes transient). (4) If all fail with matching environments, send IT this ticket: 
Request: Dataverse security role update for my user in the
[environment name] environment (Copilot Studio / Power Platform).
I am building an approved marketing demo. When my Copilot Studio
agent calls an agent flow as a tool from Microsoft 365 Copilot, I
receive: "User does not have sufficient permissions in Microsoft
Dataverse for flow '[flow id from the error]'."
Please grant the Basic User role plus maker/run privileges for
flows in this environment (or advise the correct role).
(5) Demo-safe fallback needing no flow connection: add to the agent's instructions — "When asked for the final page, output the complete HTML document in a single code block." Save that response as a .html file and open it in a browser: same artifact, zero Dataverse. The flow stays in the story as the assembly step, demonstrated from Studio.
Agent doesn't appear in the Copilot app. Publishing didn't include the Microsoft 365 Copilot channel — go back to the agent's publish/channels settings and enable it; allow a few minutes to appear.
Image generation refuses the brief. Usually a compliance filter on financial imagery or people. Remove references to money outcomes and specific people; describe scene, mood, and style only. The instructions already ban real people, logos, and in-image text for this reason.
Deck ignores your template look. The template wasn't the open, active presentation when you clicked Copilot. Open the template file first, then Create from file.
